# Introduction

Home Server is an ongoing engineering project built to explore software
architecture, systems programming, networking, and self-hosted
infrastructure through a single evolving platform.

Rather than creating isolated projects for every technology, the objective
is to build an ecosystem where every new service solves a real problem while
also introducing an opportunity to learn something new.

The project is intentionally developed in small iterations. Every new
component expands the platform without replacing previous work, allowing the
architecture to mature over time instead of being designed all at once.

Although the project currently focuses on infrastructure and backend
development, its long-term goal is much broader: to provide a complete
platform that can be managed from desktop and mobile devices through a
consistent user experience.

At the time of writing, the networking foundation has been established, the
initial Rust architecture is under active development, and the first shared
libraries are beginning to define the conventions that future services will
follow.

# Why Build a Home Server?

Many software projects exist only to demonstrate a technology.

Home Server was created for a different reason.

The objective is not to build a web server, a dashboard, or an API in
isolation. Instead, the goal is to create a real environment where every
technology contributes to a larger system.

Learning Rust becomes more meaningful when the language powers real
services.

Learning Linux becomes more valuable when the operating system hosts those
services.

Learning networking becomes practical when the server itself must be
accessible and reliable.

Instead of treating every technology as an independent tutorial, Home Server
connects them into a single engineering project.

# Project Vision

The long-term vision is to transform a single physical computer into a
personal platform capable of hosting multiple services that communicate
through shared libraries and common architectural conventions.

The server itself is only one part of the system.

Around it, several layers are expected to evolve independently while sharing
the same infrastructure.

```text
                    React Dashboard
                           │
               Native Android App (Future)
                           │
                   ASP.NET Communication
                           │
─────────────────────────────────────────────────────
                Shared Rust Infrastructure
─────────────────────────────────────────────────────
 Logging • Configuration • Responses • Utilities
─────────────────────────────────────────────────────
 Docker • Linux • Networking • Storage • Monitoring
─────────────────────────────────────────────────────

```

Each layer has a clearly defined responsibility.

The infrastructure provides the environment.

Rust implements reusable platform services.

.NET exposes communication endpoints.

React becomes the primary administration interface.

A native Android application is planned for future iterations, providing a
mobile experience optimized for monitoring and lightweight administrative
tasks.

Although several of these layers are still under development, defining their
responsibilities early has influenced many architectural decisions already
present in the project.

# Current Status

Home Server should not be considered a finished platform.

At this stage, most effort has been invested in building the foundation
required for future services rather than implementing user-facing features.

Current progress includes:

- Initial Linux server configuration.
- Static network configuration.
- Remote administration through SSH.
- Initial Rust project architecture.
- Shared infrastructure modules.
- Logging abstraction.
- Standardized API response models.

Future work will gradually introduce additional backend services, web
interfaces and mobile applications without requiring major architectural
changes.

# Infrastructure

Before writing application code, the server itself needed to become a stable
development environment.

This first phase focused entirely on preparing the operating system,
networking configuration and remote administration.

Although these tasks do not produce visible features, they establish the
foundation upon which every future service depends.

# Network Configuration

One of the earliest goals was ensuring that the server could always be
reached from other devices on the local network.

Using dynamically assigned IP addresses would eventually introduce
unnecessary friction during development, as the server's address could change
after a reboot or router lease renewal.

Instead, the server was configured with a static IP address, allowing every
future application and development environment to communicate with a stable
network endpoint.

While simple, this decision removes an entire category of operational
problems before additional services are introduced.

# Remote Administration

A server should not require physical access every time a configuration
changes.

For that reason, SSH became the primary administration mechanism from the
very beginning.

Once configured, the server could be managed directly from another computer,
making software development, maintenance and experimentation significantly
more practical.

This workflow closely resembles real production environments, where servers
are rarely administered through a directly connected monitor.

# Infrastructure Philosophy

One recurring design principle throughout the project is to build reusable
infrastructure before building individual features.

Rather than allowing every future application to implement its own logging,
configuration management or communication models, common functionality is
being extracted into shared libraries from the beginning.

Although this approach requires more work initially, it significantly
reduces duplication as the platform grows.

The goal is not to optimize for today's requirements, but to create an
architecture capable of supporting years of incremental development.

# Looking Ahead

With the networking foundation established and the development environment in
place, attention shifts toward the software platform itself.

The next phase introduces the Rust core responsible for providing shared
infrastructure across the platform, beginning with one of the most important
cross-cutting concerns in any distributed system: logging.

# Building the Platform Core

With the server infrastructure in place, development shifted toward the
software architecture itself.

Rather than implementing complete services immediately, the first objective
was to define the common building blocks that every future component would
share.

Logging, configuration, response models and reusable utilities are examples
of concerns that should not be reimplemented by every service.

Instead, they belong to the platform itself.

This philosophy has influenced the project from its earliest stages.

# Why Rust?

Rust was selected primarily as a learning objective.

Although languages like C# would have allowed faster initial development,
Rust introduces concepts that encourage thinking differently about software
design: ownership, borrowing, explicit lifetimes and stronger guarantees
around memory safety.

More importantly, the language fits naturally within the long-term vision of
the project.

Backend services running continuously benefit from predictable performance,
low memory usage and compile-time guarantees, making Rust an excellent
candidate for platform infrastructure.

Learning these concepts within a real project provides much stronger
motivation than isolated exercises.

# Shared Infrastructure

Several modules already exist even before any business logic has been
implemented.

These modules define conventions rather than functionality.

Instead of solving individual application problems, they provide common
behavior shared across the entire platform.

Current shared modules include:

- Logging
- Error definitions
- Configuration
- API response models
- Utility functions

The expectation is that future services will import these modules rather
than creating independent implementations.

# Designing the Logging System

Logging is one of the first cross-cutting concerns introduced into the
project.

Instead of calling the tracing library directly throughout the codebase,
the objective is to isolate that implementation behind an internal API.

This provides two important advantages.

First, application code does not depend directly on a specific logging
library.

Second, replacing or extending the logging engine becomes significantly
easier in the future.

The application knows that logging exists.

It does not need to know how logging is implemented.

# Builder Pattern

Configuration is performed through a builder rather than constructing
configuration objects manually.

```rust
let config = LoggingBuilder::new()
    .level(LogLevel::Debug)
    .timestamp(true)
    .colors(true)
    .build();
```

This approach makes configuration expressive while preventing partially
initialized objects from appearing throughout the application.

Additional options can be introduced later without affecting existing code.

# Separating Configuration

Instead of embedding logging options directly inside the logger itself,
configuration lives independently.

```rust
pub struct LoggingConfig {
    pub level: LogLevel,
    pub timestamp: bool,
    pub colors: bool,
}
```

Keeping configuration separate improves readability while making future
serialization or configuration-file support much easier to introduce.

The logger becomes responsible for behavior.

The configuration object becomes responsible for state.

# Abstracting the Logging Engine

Internally, the current implementation relies on the `tracing` ecosystem.

However, this dependency is intentionally hidden behind conversion layers and
internal abstractions.

One small example is the conversion between the project's own logging levels
and the levels expected by `tracing`.

```rust
impl From<LogLevel> for Level {
    fn from(level: LogLevel) -> Self {
        match level {
            LogLevel::Error => Level::ERROR,
            LogLevel::Warn => Level::WARN,
            LogLevel::Info => Level::INFO,
            LogLevel::Debug => Level::DEBUG,
            LogLevel::Trace => Level::TRACE,
        }
    }
}
```

Although relatively simple, this separation ensures that the rest of the
application depends only on platform types rather than external libraries.

# Error Handling

Initialization failures are represented using explicit error types.

```rust
pub enum LoggingError {
    AlreadyInitialized,
    InvalidConfiguration,
    InitializeFailed,
}
```

Rather than exposing generic strings, these error types communicate the
expected failure modes directly through the type system.

As additional infrastructure components are introduced, they are expected to
follow the same pattern.

# Shared Response Models

Another piece of infrastructure already exists despite there being few API
endpoints today.

Instead of allowing every controller or service to define independent
response structures, standardized response envelopes have been introduced.

Successful responses follow a common structure.

```rust
pub struct ApiResponse<T> {
    success: bool,
    timestamp: DateTime<Utc>,
    data: T,
}
```

Errors follow the same philosophy.

```rust
pub struct ApiError {
    success: bool,
    timestamp: DateTime<Utc>,
    error: ErrorResponse,
}
```

Even simple endpoints, such as future health checks, will benefit from these
shared models.

```rust
pub struct HealthResponse {
    pub status: String,
}
```

Although these abstractions appear small, they establish consistency before
multiple services begin exposing APIs.

# Architecture Before Features

At this stage, the project intentionally contains more infrastructure than
business functionality.

That balance is expected.

Rather than building several independent services and attempting to unify
them later, the project establishes shared conventions first.

As new services appear, they inherit those conventions automatically instead
of introducing their own implementations.

This approach slows down early development slightly but significantly
reduces architectural fragmentation as the platform grows.

# Looking Ahead

With the shared infrastructure beginning to stabilize, the next step is to
connect these components to actual applications.

The following phase introduces the communication layer, web interface and
future mobile client that will allow the platform to be managed without
direct interaction with the server itself.

# The Communication Layer

Although the Rust core is responsible for the platform's internal
infrastructure, it is not intended to communicate directly with user
interfaces.

A dedicated .NET service is planned to act as the communication layer
between clients and the backend.

Rather than exposing internal implementation details, this layer will
provide a stable API capable of serving multiple clients simultaneously.

```text
 React Dashboard
        │
 Android Application
        │
────────▼────────
 ASP.NET Core API
────────▼────────
 Rust Services
```

Separating communication from the platform core keeps responsibilities
well-defined.

Rust focuses on platform services.

.NET focuses on exposing those services through HTTP APIs.

The two layers evolve independently while communicating through clearly
defined contracts.

# Why .NET?

The decision to introduce .NET is based both on familiarity and on the role
it already plays in my professional work.

Using ASP.NET Core allows new APIs to be developed quickly while taking
advantage of a mature ecosystem for authentication, routing, dependency
injection and API documentation.

Rather than replacing Rust, .NET complements it.

Each technology is used where it provides the greatest value.

# React Dashboard

Managing infrastructure through SSH works well during development, but it
does not scale as the platform grows.

One of the project's primary goals is to make administration accessible
through a modern web interface.

The React dashboard is intended to become the central control panel for the
entire platform.

Rather than exposing isolated pages for each service, the dashboard should
present the server as a single system.

Potential features include:

- Service monitoring
- Configuration management
- Log visualization
- Health monitoring
- Performance metrics
- Container administration
- Future services added to the platform

Because every backend service follows shared conventions, the dashboard can
provide a consistent experience regardless of which component it is
interacting with.

# Mobile Administration

Although the React dashboard is expected to work well across desktop and
mobile browsers, a dedicated Android application is planned for future
iterations.

The intention is not to wrap the website inside a WebView.

Instead, the application will be developed natively using Kotlin.

A native application provides better integration with Android features such
as notifications, widgets and background services while offering an
interface specifically designed for quick administrative tasks.

Examples include:

- Checking server health.
- Viewing recent logs.
- Restarting services.
- Receiving alerts.
- Monitoring resource usage.

This application is considered a long-term objective and has not yet entered
active development.

# Development Philosophy

Home Server is intentionally developed through small, incremental
iterations.

Instead of attempting to design every service before writing code, each new
component introduces only the infrastructure required for the next step.

This approach produces continuous progress while allowing architectural
decisions to evolve naturally as new requirements appear.

The current logging infrastructure is an example of this philosophy.

It exists not because logging was urgently required, but because every
future service will eventually depend on it.

Building that shared foundation early prevents duplicated work later.

# Lessons Learned

Even in its current state, Home Server has already influenced the way I
approach software architecture.

One of the most important lessons has been recognizing the value of building
shared infrastructure before individual features.

While this often delays visible progress, it dramatically simplifies future
development by encouraging consistency across the entire platform.

The project has also reinforced the importance of clearly separating
responsibilities.

Networking, infrastructure, backend services, communication APIs and user
interfaces each solve different problems.

Treating them as independent layers makes the overall system easier to
understand, maintain and extend.

Finally, working on a long-term project has highlighted the importance of
designing for evolution rather than completion.

The objective is not to finish the platform as quickly as possible.

The objective is to create an environment where new ideas can be introduced
without requiring the architecture to be redesigned every few months.
