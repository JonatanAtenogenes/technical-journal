# Resumen

Technical Journal es una plataforma de documentación diseñada para presentar proyectos de ingeniería de software, infraestructura y análisis de datos como casos de estudio técnicos.

En lugar de mostrar proyectos como simples demostraciones o repositorios de GitHub, cada entrada documenta el proceso completo de ingeniería detrás de su desarrollo: el problema original, el contexto de negocio, la arquitectura, la implementación, las decisiones técnicas, los compromisos asumidos, las distintas iteraciones y las lecciones aprendidas.

La plataforma busca cerrar la brecha entre un portafolio tradicional y el trabajo real de un ingeniero. Mientras un portafolio responde a la pregunta *«¿Qué has construido?»*, Technical Journal intenta responder otra igualmente importante: *«¿Cómo lo construiste y por qué tomaste esas decisiones?»*

Cada proyecto se trata como documentación viva. Los casos de estudio evolucionan junto con el software que describen, permitiendo incorporar nuevas funcionalidades, cambios de arquitectura y aprendizajes a medida que el proyecto continúa creciendo, en lugar de quedar congelados en el momento de su primera versión.

En última instancia, el objetivo es que el journal se convierta en un cuaderno de ingeniería a largo plazo, más que en una simple colección de proyectos terminados.

# ¿Por Qué Construirlo?

Los portafolios tradicionales son excelentes para mostrar tecnologías y proyectos terminados, pero rara vez consiguen transmitir el razonamiento que hay detrás de una solución técnica.

Un reclutador puede identificar React, .NET, Docker, SQL o Python dentro de un proyecto y, aun así, no comprender por qué se eligieron esas tecnologías, qué alternativas se evaluaron, qué restricciones influyeron en la arquitectura o cómo evolucionó la implementación conforme aparecieron nuevos requisitos.

Con frecuencia, esas decisiones representan la parte más valiosa de un proyecto, pero difícilmente caben dentro de una tarjeta de portafolio o en un README.

Technical Journal nació precisamente para documentar ese contexto que normalmente se pierde.

El portafolio funciona como el punto de entrada, ofreciendo una visión general de cada proyecto y facilitando su descubrimiento. Technical Journal lo complementa actuando como la documentación técnica detrás de cada uno, donde es posible profundizar en la implementación, las decisiones arquitectónicas y la evolución del desarrollo sin comprometer la simplicidad del portafolio.

Separar ambas responsabilidades permite que cada plataforma cumpla mejor su propósito. El portafolio permanece limpio, accesible y orientado a una primera impresión, mientras que el journal ofrece el nivel de profundidad técnica esperado en un caso de estudio de ingeniería.

# Principios de Diseño

Diversos principios guían cada decisión de arquitectura e implementación dentro de la plataforma.

## La Documentación Como Prioridad

La documentación se considera un producto principal del desarrollo, no una tarea que se realiza una vez terminado el código.

La aplicación existe para facilitar la escritura y lectura de documentación técnica. Cada nueva funcionalidad se evalúa en función de si mejora el proceso de documentar proyectos de ingeniería o la experiencia de quienes los leen.

## Separación de Responsabilidades

El contenido, la presentación, el enrutamiento y los metadatos permanecen intencionalmente separados.

La documentación técnica nunca debería contener lógica de presentación, y los componentes de la interfaz no deberían conocer la estructura interna de un artículo. Esta separación permite que tanto el modelo de contenido como la interfaz evolucionen de manera independiente.

## Estático Por Defecto

Siempre que es posible, el contenido se genera durante el proceso de compilación.

Evitar bases de datos y APIs en tiempo de ejecución simplifica el despliegue, mejora la confiabilidad y permite aprovechar las ventajas de rendimiento de los sitios estáticos sin sacrificar flexibilidad para la escritura técnica.

## Arquitectura Incremental

Las funcionalidades se incorporan mediante pequeñas iteraciones independientes en lugar de grandes abstracciones diseñadas desde el inicio.

En vez de intentar anticipar todos los requisitos futuros, la plataforma evoluciona conforme aparecen necesidades reales. Las abstracciones solo se introducen cuando un flujo de trabajo manual demuestra que resuelven un problema existente.

Esta filosofía mantiene la complejidad bajo control y permite que la arquitectura crezca de forma natural.

## Consistencia Antes Que Flexibilidad

Todos los casos de estudio siguen una estructura común.

Aunque cada proyecto sea diferente, mantener una organización consistente hace que la documentación sea más sencilla de escribir y mucho más fácil de leer. El lector siempre sabe dónde encontrar las decisiones de arquitectura, los detalles de implementación y las lecciones aprendidas, independientemente del proyecto que esté consultando.

## Lectura Cómoda Para Documentación Extensa

La plataforma está diseñada para leer artículos técnicos largos, no para consumir publicaciones breves como las de un blog.

La navegación, la tipografía, los espacios, el indicador de progreso y el diseño responsivo están pensados para reducir la fricción durante sesiones de lectura prolongadas tanto en computadoras como en dispositivos móviles.

# Arquitectura

Technical Journal está construido con Next.js utilizando App Router y React Server Components.

La aplicación fue concebida como una plataforma completamente estática. Cada proyecto se descubre directamente desde el sistema de archivos durante la compilación, eliminando la necesidad de bases de datos, APIs o un sistema de gestión de contenido, mientras se mantiene un proceso de despliegue simple.

Cada caso de estudio combina metadatos estructurados con documentos MDX, permitiendo que la documentación permanezca junto al código fuente sin perder una experiencia de escritura limpia y cómoda. Al vivir dentro del mismo repositorio, la documentación evoluciona mediante el mismo flujo de trabajo basado en Git que el resto del software.

La mayor parte del renderizado se realiza mediante React Server Components, reduciendo al mínimo el JavaScript enviado al cliente. Las funcionalidades interactivas —como el indicador de progreso de lectura, la navegación, el cambio de tema y otros comportamientos de la interfaz— permanecen aisladas en componentes de cliente específicos.

Esta arquitectura prioriza la simplicidad, la mantenibilidad y el rendimiento, al mismo tiempo que proporciona una base suficientemente flexible para que la plataforma continúe creciendo sin requerir cambios estructurales importantes.

# Modelo de Contenido

Cada proyecto documentado reside dentro de su propio directorio en la carpeta de contenido.

En lugar de almacenar toda la información en un único archivo, las responsabilidades se dividen según el tipo de información que representan.

* `metadata.ts` contiene la información estructurada utilizada por la aplicación, incluyendo títulos, descripciones, estado de publicación, tecnologías, etiquetas, imágenes de portada, fechas y futuros metadatos para SEO.
* `index.mdx` contiene el artículo técnico propiamente dicho. Su contenido se centra exclusivamente en la documentación, sin incluir lógica de presentación, navegación o comportamiento específico de la aplicación.

Esta separación permite que la interfaz consuma información estructurada mientras la documentación permanece completamente independiente de la implementación de la plataforma.

Conforme el proyecto evolucione, esta organización facilitará la incorporación de nuevos campos de metadatos o mejoras en la interfaz sin necesidad de modificar los artículos existentes.

# Experiencia de Usuario

La experiencia de lectura es uno de los principales objetivos del proyecto.

En lugar de presentar la documentación técnica como grandes bloques de texto, la interfaz proporciona continuamente información contextual que ayuda al lector a navegar artículos extensos sin perder su ubicación.

Actualmente, la plataforma incluye:

* Un encabezado dedicado para cada caso de estudio.
* Una tabla de contenidos generada automáticamente.
* Indicadores de progreso de lectura.
* Navegación fija en escritorio.
* Diseños responsivos con enfoque *mobile-first*.
* Etiquetas de tecnologías.
* Temas claro y oscuro.

Estos elementos trabajan en conjunto para reducir la carga cognitiva durante la lectura de documentación extensa. En lugar de obligar al lector a buscar manualmente una sección específica, la interfaz comunica constantemente dónde se encuentra, cuánto contenido queda por recorrer y cómo está organizado el documento.

El resultado es una experiencia más cercana a la lectura de documentación técnica profesional que a la de un blog tradicional.

# Filosofía de Desarrollo

La plataforma sigue un proceso de desarrollo iterativo.

En lugar de diseñar todos los sistemas antes de escribir una sola línea de código, cada funcionalidad comienza con la implementación más sencilla capaz de aportar valor. Las abstracciones, automatizaciones y mejoras arquitectónicas se introducen únicamente cuando la experiencia práctica demuestra que realmente son necesarias.

Este enfoque reduce la complejidad innecesaria y permite que las decisiones de arquitectura surjan como consecuencia del uso real de la aplicación, y no de suposiciones.

## Construyendo Primero la Interfaz

Antes de incorporar MDX, modelos de metadatos, carga de contenido desde el sistema de archivos o descubrimiento automático de proyectos, toda la interfaz fue desarrollada utilizando datos simulados.

Esto permitió validar el espaciado, la tipografía, los diseños responsivos, los patrones de navegación y el flujo general de lectura sin tener que resolver simultáneamente problemas relacionados con el modelo de datos.

Al separar el diseño de la interfaz de la implementación del contenido, el modelo de datos pudo diseñarse posteriormente sobre necesidades reales en lugar de hipótesis.

## Incorporando Contenido Real

Una vez que la interfaz demostró ser estable, la aplicación comenzó a consumir metadatos reales y documentos MDX.

Como la capa de presentación ya había sido validada, la transición desde datos simulados requirió muy pocos cambios estructurales. Los componentes existentes se adaptaron de forma natural a las nuevas fuentes de información, confirmando que las decisiones iniciales de arquitectura habían producido una base flexible.

Esta transición incremental redujo considerablemente la complejidad de la implementación y reforzó uno de los principios fundamentales del proyecto: resolver un problema a la vez, validar la solución y solo entonces avanzar hacia la siguiente capa de abstracción.

# Evolución del Proyecto

Uno de los objetivos originales de Technical Journal fue documentar el trabajo de ingeniería conforme evoluciona, en lugar de describir únicamente el resultado final.

Por esa razón, este caso de estudio sigue la misma filosofía que la propia plataforma. En lugar de permanecer estático después de la primera versión, crece junto con la aplicación, documentando los cambios importantes de arquitectura, los detalles de implementación, los compromisos asumidos y las lecciones aprendidas conforme se incorporan nuevas funcionalidades.

Cada iteración importante se trata como un hito de ingeniería independiente.

# Iteración 1 — Construyendo la Base

La primera iteración estuvo enfocada en validar la idea central de la plataforma: reemplazar las páginas tradicionales de proyectos por casos de estudio técnicos.

En esta etapa, la prioridad no era añadir funcionalidades, sino establecer una arquitectura capaz de soportar documentación técnica extensa sin dejar de ser lo suficientemente sencilla para mantenerla a largo plazo.

Desde el principio se tomaron varias decisiones fundamentales.

El contenido de los proyectos viviría directamente dentro del repositorio en lugar de almacenarse en una base de datos.

Los artículos se escribirían utilizando MDX para mantener la documentación cerca del código fuente sin renunciar a una experiencia de escritura limpia.

Los metadatos estructurados permanecerían separados del contenido del artículo para que la aplicación pudiera consumir información de los proyectos sin acoplarse a la documentación escrita.

La navegación se generaría automáticamente a partir de los encabezados del documento, eliminando la necesidad de mantener una estructura de navegación independiente.

Finalmente, toda la plataforma sería completamente estática, evitando infraestructura innecesaria y maximizando tanto el rendimiento como la simplicidad del despliegue.

Aunque estas decisiones parecían relativamente pequeñas, terminaron definiendo prácticamente todas las funcionalidades implementadas posteriormente.

En lugar de intentar anticipar necesidades futuras, la arquitectura inicial se limitó a resolver los problemas que realmente existían en ese momento.

Esto permitió construir una base limpia que las siguientes iteraciones pudieron extender sin requerir reescrituras importantes.

# Iteración 2 — Internacionalización

Conforme la plataforma fue madurando, el soporte para múltiples idiomas se convirtió en la primera gran evolución arquitectónica del proyecto.

A diferencia de añadir una funcionalidad aislada, la internacionalización afectó prácticamente todas las capas de la aplicación: el sistema de rutas, la organización del contenido, los metadatos, la navegación e incluso algunos aspectos de la experiencia de lectura.

En lugar de tratar la traducción como una característica añadida al final del desarrollo, el objetivo fue convertir el soporte para múltiples idiomas en una parte fundamental de la arquitectura.

## ¿Por Qué Agregar Internacionalización?

La audiencia de la plataforma no se limita a desarrolladores que hablan inglés.

Si el propósito de Technical Journal es explicar decisiones de ingeniería, esas explicaciones deberían estar disponibles en el idioma con el que el lector se sienta más cómodo.

Una simple opción para cambiar el idioma desde el cliente habría permitido traducir la interfaz visible, pero no habría generado URLs independientes por idioma, páginas indexables por separado ni metadatos localizados.

Por ello, la implementación se diseñó para que cada idioma fuera una versión completa de la aplicación y no simplemente una traducción aplicada sobre la interfaz existente.

## Enrutamiento Dependiente del Idioma

La plataforma adoptó un sistema de rutas localizadas utilizando **next-intl**.

El inglés permanece como idioma principal, mientras que el español se publica bajo su propio prefijo de URL.

```text
/en
```

```text
/es
```

Todas las páginas —la página principal, el listado de proyectos y cada caso de estudio— están disponibles mediante su propia ruta localizada.

Este enfoque aporta varias ventajas.

Cada idioma puede ser indexado de forma independiente por los motores de búsqueda.

Las páginas pueden compartirse directamente en un idioma específico.

Los React Server Components reciben el idioma activo durante el renderizado sin necesidad de sincronización adicional desde el cliente.

Como la aplicación ya era completamente estática, la internacionalización simplemente amplió el proceso de generación existente.

`generateStaticParams` ahora produce todas las combinaciones posibles entre proyectos e idiomas soportados, permitiendo que cada versión localizada continúe generándose durante la compilación.

## Un Modelo de Contenido Bilingüe

La incorporación de múltiples idiomas obligó a replantear la organización del contenido.

Originalmente, cada proyecto estaba compuesto únicamente por un archivo de metadatos y un documento MDX.

Con la internacionalización, parte de esa información necesitaba traducirse mientras que otra permanecía exactamente igual sin importar el idioma.

En lugar de duplicar completamente la definición de cada proyecto, el modelo de contenido se dividió en tres capas independientes.

```text
project/
├── meta.ts
├── en/
│   ├── metadata.ts
│   └── index.mdx
└── es/
    ├── metadata.ts
    └── index.mdx
```

Cada una de ellas tiene una responsabilidad claramente definida.

* `meta.ts` almacena la información independiente del idioma, como el estado de publicación, las tecnologías, las imágenes de portada, las etiquetas y los años.
* `metadata.ts` contiene únicamente los campos traducibles, como el título, la categoría y la descripción.
* `index.mdx` contiene el artículo correspondiente a cada idioma.

Separar la información estructural del contenido traducible eliminó duplicación innecesaria y simplificó considerablemente el mantenimiento.

Agregar un nuevo idioma ya no implica copiar grandes objetos de metadatos, sino únicamente traducir la información que realmente cambia.

## Localizando la Interfaz

La internacionalización va más allá del contenido de los artículos.

Los textos de navegación, botones, encabezados de sección, etiquetas y demás elementos de la interfaz fueron extraídos a diccionarios de traducción.

En lugar de organizar las traducciones por página, estos diccionarios siguen la misma estructura que los componentes de la aplicación.

Un componente compartido, como el encabezado, mantiene sus propias traducciones, mientras que los componentes específicos de una página contienen únicamente los textos que utilizan.

Esta organización refleja directamente la arquitectura del proyecto y facilita localizar cada cadena de texto conforme la aplicación continúa creciendo.

## Navegación al Portafolio Según el Idioma

Technical Journal y el portafolio fueron concebidos como dos plataformas complementarias.

Antes de implementar la internacionalización, el enlace hacia el portafolio apuntaba siempre al mismo sitio, independientemente del idioma desde el cual navegara el usuario.

Ahora ese enlace se resuelve dinámicamente.

Un visitante que navega el journal en inglés es dirigido automáticamente a la versión en inglés del portafolio, mientras que un lector en español accede directamente a la versión localizada.

Aunque se trata de un cambio relativamente pequeño, elimina cambios innecesarios de contexto y mantiene una experiencia consistente entre ambas plataformas.

## Conservando la Posición de Lectura

Durante la implementación apareció un problema inesperado relacionado con la experiencia de usuario.

Al cambiar de idioma mientras se leía un artículo, la página traducida siempre comenzaba desde el inicio del documento.

Inicialmente parecía suficiente conservar el fragmento (`#hash`) de la URL.

Sin embargo, los encabezados traducidos generan identificadores distintos, haciendo que los enlaces internos dejen de coincidir entre idiomas.

La solución consistió en registrar la posición del lector utilizando el índice del encabezado dentro de la tabla de contenidos generada automáticamente.

Justo antes de navegar, se almacena el índice del encabezado actualmente visible.

Una vez que la versión traducida termina de renderizarse, el componente encargado del progreso de lectura recupera ese índice y desplaza automáticamente la página hasta el encabezado equivalente.

Este enfoque asume que ambas traducciones mantienen la misma estructura y el mismo orden de secciones, una restricción perfectamente razonable considerando que ambas versiones buscan comunicar exactamente el mismo contenido técnico.

El resultado es un cambio de idioma prácticamente transparente, permitiendo continuar la lectura desde el mismo punto donde se encontraba el usuario.

## Problemas Encontrados

Durante la implementación aparecieron varios inconvenientes cuya solución no resultó evidente únicamente consultando la documentación.

### Conflicto con el Root Layout

Como el idioma activo determina el atributo `lang` del documento HTML, ya no era posible mantener un layout raíz independiente por encima de las rutas localizadas.

La solución consistió en integrar el layout principal dentro del layout específico de cada idioma para poder resolver el locale antes de renderizar el elemento `<html>`.

### Middleware Renombrado a Proxy

Mientras se desarrollaba esta funcionalidad, Next.js 16 reemplazó la convención `middleware.ts` por `proxy.ts`.

Aunque el comportamiento del sistema de rutas permaneció exactamente igual, fue necesario actualizar tanto el nombre del archivo como el nombre de la función exportada.

Fue un cambio relativamente pequeño, pero ilustró cómo la evolución del framework puede afectar una implementación incluso cuando la lógica de negocio permanece intacta.

### Un Hook Sin Invocar

Uno de los errores más sutiles apareció al pasar una referencia al hook en lugar de invocarlo.

En vez de utilizar `usePathname()`, accidentalmente se pasó `usePathname`.

Como consecuencia, el router serializó la propia función dentro de la URL de destino, generando rutas completamente inválidas a pesar de que el código compilaba correctamente.

La corrección fue sencilla una vez identificado el problema, pero recordó que no todos los errores de ejecución vienen acompañados de mensajes claros del compilador.

### Re-renderizados Infinitos

Otro inconveniente surgió durante la implementación de la restauración de la posición de lectura.

Un arreglo derivado se recalculaba en cada render y era utilizado como dependencia dentro de un `useEffect`, provocando un ciclo infinito de renderizados.

Las asignaciones repetidas terminaron agotando la memoria disponible para Node durante el desarrollo.

La solución consistió en memorizar el cálculo mediante `useMemo`, estabilizando las dependencias y eliminando el ciclo.

### Caché Obsoleta de Turbopack

Después de varios cambios estructurales —como reorganizar las rutas bajo un segmento `[locale]` y modificar la estructura de los layouts— el servidor de desarrollo comenzó a comportarse de forma inconsistente.

Eliminar el directorio `.next` obligó a Turbopack a reconstruir completamente su caché, resolviendo inmediatamente el problema.

Aunque simple, esta terminó convirtiéndose en una de las primeras acciones de diagnóstico después de realizar refactorizaciones importantes.

### Inconsistencia en las Claves de Almacenamiento

La restauración de la posición de lectura dejó de funcionar porque se utilizaban claves distintas al escribir y leer datos desde `sessionStorage`.

Más allá de corregir el error puntual, todas las claves fueron centralizadas en un único archivo de constantes compartidas.

De esta forma, futuras inconsistencias pasarían a detectarse como errores de TypeScript en lugar de convertirse en fallos silenciosos en tiempo de ejecución.

## Lecciones Aprendidas

La internacionalización terminó siendo mucho más que un sistema de traducción.

Su implementación obligó a replantear el sistema de rutas, la organización del contenido, la navegación, la responsabilidad de los metadatos e incluso algunos aspectos de la experiencia de usuario.

Quizá el aprendizaje más importante fue confirmar el valor de separar claramente la información estructural del contenido traducible.

Gracias a que los metadatos, los artículos y la lógica de la aplicación ya tenían responsabilidades bien definidas, la incorporación de múltiples idiomas consistió principalmente en extender la arquitectura existente en lugar de reemplazarla.

Más allá de la internacionalización, esta iteración reforzó una idea que ha acompañado al proyecto desde sus primeras versiones: los cambios arquitectónicos importantes resultan mucho más sencillos de implementar cuando el sistema ha sido diseñado desde el principio con límites claros entre sus responsabilidades y una filosofía de evolución incremental.

# Iteración 3 — Renderizado Profesional de Bloques de Código

A medida que crecía el número de case studies, los bloques de código planos se volvieron una brecha visible entre Technical Journal y las plataformas de documentación que lo inspiraron. Los walkthroughs con múltiples archivos, comandos de terminal y diffs se veían todos idénticos, sin syntax highlighting, sin contexto de archivo y sin forma de distinguir líneas agregadas o eliminadas.

Esta iteración se enfocó en cerrar esa brecha sin comprometer la arquitectura static-first de la plataforma ni introducir una dependencia que compitiera con el sistema de diseño existente.

## Por Qué Agregar Esto

Una plataforma de documentación inspirada en Vercel Docs, GitHub Docs y Stripe Docs no puede depender de bloques de código sin estilo. El código no es incidental en un case study—frecuentemente es la parte más importante de la explicación, y merece el mismo nivel de cuidado visual que la tipografía que lo rodea.

Más allá de lo estético, se habían acumulado varias necesidades prácticas a lo largo de los artículos existentes: distinguir a qué archivo pertenecía un snippet, resaltar líneas específicas siendo discutidas, y hacer que los ejemplos de código largos fueran más fáciles de escanear sin salir del flujo de lectura.

## Eligiendo Expressive Code

Se eligió Expressive Code sobre alternativas más ligeras como `rehype-pretty-code` porque provee estas capacidades de fábrica en lugar de requerir que se construyan manualmente: frames con nombre de archivo, line numbers, resaltado de diffs, secciones colapsables y ventanas estilo terminal.

Aunque Expressive Code se originó en el ecosistema de Astro, incluye una integración agnóstica de framework, `rehype-expressive-code`, construida específicamente para cualquier pipeline basado en rehype—incluyendo el que ya potencia el renderizado de MDX de esta plataforma.

## Integrando a Través de rehype, No de next.config

La mayoría de la documentación de `rehype-expressive-code` asume una integración a través de `@next/mdx`, registrando el plugin dentro de `next.config.js`. Technical Journal no usa `@next/mdx`; el MDX se compila en tiempo de request mediante `next-mdx-remote/rsc`, directamente dentro del Server Component que renderiza cada case study.

El plugin se registró en su lugar en el array `rehypePlugins` que se le pasa a `MDXRemote`, junto a los ya existentes `remark-gfm` y `rehype-slug`. Como esta compilación ocurre en ejecución normal de Node.js en lugar de a través del loader de MDX de Turbopack basado en Rust, pasar una referencia de función directamente no introdujo ninguna de las restricciones de serialización que aplican al enfoque de `next.config.js`.

Esta distinción fue importante: significó que el pipeline de MDX existente pudo extenderse sin introducir un segundo camino de compilación basado en configuración en paralelo.

## Consistencia con el Sistema de Diseño Existente

El principal riesgo de adoptar una librería de terceros para el renderizado de código era la inconsistencia visual—bloques de código que se vieran como un widget insertado en lugar de parte del sitio.

Dos decisiones de configuración resolvieron esto directamente.

El cambio de tema se ligó a la implementación de dark mode ya existente en la plataforma. Por defecto, Expressive Code usa una media query `prefers-color-scheme`, lo cual habría ignorado el toggle manual ya construido con `next-themes`. Deshabilitar `useDarkModeMediaQuery` y configurar `themeCssSelector` para calzar con la estrategia de clase `.dark` del sitio hizo que los bloques de código cambiaran de tema en sincronía con el resto de la interfaz, en lugar de seguir al sistema operativo de forma independiente.

Los style overrides se mapearon a las propias custom properties de CSS del proyecto—border radius, color de borde y familia tipográfica—en lugar de los valores por defecto de Expressive Code. Esto mantuvo los bloques de código anclados visualmente a los mismos design tokens usados en el resto de los componentes.

## Lecciones Aprendidas

Esta iteración no requirió cambios en el modelo de contenido ni en el flujo de autoría de MDX. Los bloques de código existentes en cada case study, en ambos idiomas, obtuvieron syntax highlighting, frames con nombre de archivo y line numbers automáticamente la siguiente vez que se renderizaron—ningún artículo necesitó ser reescrito.

También reforzó un patrón ya establecido durante la internacionalización: como el pipeline de renderizado de MDX tenía un único punto de entrada bien definido, extenderlo con una nueva capacidad significó agregar un plugin a un array, en lugar de tocar cada lugar donde se renderiza contenido.

# Decisiones Técnicas

Aunque la plataforma ha evolucionado mediante diferentes iteraciones, varias decisiones arquitectónicas se han mantenido constantes desde el inicio. Estas decisiones continúan definiendo tanto la implementación como la forma en la que se crean nuevos casos de estudio.

## Contenido Estático en Lugar de un CMS

Technical Journal evita intencionalmente utilizar bases de datos o sistemas tradicionales de gestión de contenido.

Cada artículo vive junto al código fuente de la aplicación, permitiendo que la documentación evolucione mediante el mismo flujo de trabajo basado en Git que el software que describe.

Este enfoque simplifica el despliegue, permite revisar cambios de documentación mediante el mismo proceso utilizado para revisar código y elimina la complejidad operativa de mantener una plataforma de contenido separada.

Para un proyecto cuyo propósito principal es documentar ingeniería de software, tratar la documentación como parte del código fuente resultó ser la solución más natural.

## MDX Como Formato de Escritura

MDX fue seleccionado porque combina la simplicidad de Markdown con la posibilidad de integrar componentes React cuando sea necesario.

La mayoría de los artículos permanecen como documentos de texto técnico tradicionales, pero la plataforma conserva la capacidad de incorporar elementos interactivos como diagramas, bloques informativos, ejemplos dinámicos o futuras herramientas de visualización sin cambiar el flujo de escritura.

El objetivo no es convertir cada artículo en una experiencia interactiva por defecto, sino proporcionar esa posibilidad cuando realmente mejore la comunicación técnica.

## Tabla de Contenidos Automática

La navegación se genera directamente a partir del contenido del documento.

Durante el procesamiento de cada archivo MDX, los encabezados son extraídos, transformados en identificadores de ancla y utilizados para construir automáticamente la tabla de contenidos.

Mantener una estructura de navegación separada implicaría duplicar información y eventualmente generar inconsistencias entre el artículo y su navegación.

Al generar ambos elementos desde la misma fuente, se garantiza que permanezcan sincronizados en todo momento.

## Páginas Basadas en Metadatos

Los metadatos estructurados funcionan como la fuente principal de información de cada proyecto.

En lugar de distribuir información como títulos, descripciones, tecnologías o estados de publicación entre múltiples componentes, cada caso de estudio mantiene estos datos centralizados en archivos específicos.

Esto permite que la capa de presentación evolucione independientemente mientras mantiene una estructura de contenido predecible y fácil de mantener.

Además, este modelo facilita incorporar nuevas capacidades en el futuro, como mejoras de SEO, filtros, búsqueda o nuevas formas de presentar proyectos sin modificar la documentación existente.

## Mejora Progresiva

Un principio constante durante el desarrollo de la plataforma ha sido que la automatización debe aparecer después de validar una necesidad real.

Cada flujo de trabajo comienza con la solución más simple que aporta valor.

Solo cuando un proceso manual demuestra ser repetitivo se transforma en una abstracción reutilizable o una herramienta automatizada.

Esta filosofía ha influido en prácticamente todas las funcionalidades del proyecto: desde el descubrimiento automático de proyectos y la organización de metadatos hasta la internacionalización.

# Evolución Futura

Aunque la implementación actual proporciona una base sólida, existen varias mejoras planeadas para continuar expandiendo la plataforma.

Cada nueva funcionalidad seguirá la misma filosofía de desarrollo incremental utilizada desde el inicio del proyecto.

Las mejoras futuras incluyen:

* Búsqueda de texto completo entre proyectos.
* Tiempo estimado de lectura.
* Casos de estudio relacionados.
* Series de proyectos.
* Galerías de imágenes enriquecidas.
* Diagramas de arquitectura interactivos.
* Mejoras en la representación de bloques de código.
* Mejoras automáticas de resaltado de sintaxis.
* Optimizaciones de SEO.
* Soporte para RSS.
* Soporte para nuevos idiomas.
* Mejoras de accesibilidad.
* Sistemas de indexación más avanzados para colecciones grandes de contenido.

El objetivo no es convertir Technical Journal en una plataforma de blogs genérica, sino continuar refinándola como un entorno especializado para documentar trabajo de ingeniería.

Cada nueva capacidad debe mejorar al menos uno de tres aspectos fundamentales: la experiencia de escritura, la experiencia de lectura o la mantenibilidad a largo plazo.

# Lecciones Aprendidas

Construir Technical Journal ha reforzado varias ideas que van más allá de este proyecto específico.

La primera es que la arquitectura se valida mucho mejor mediante la implementación que mediante la especulación.

Muchas de las decisiones más importantes de la plataforma —como el modelo de contenido, la organización de metadatos y el desarrollo incremental— surgieron después de resolver problemas reales, en lugar de intentar anticipar todos los escenarios futuros.

Otro aprendizaje importante ha sido el valor de separar responsabilidades desde etapas tempranas.

Mantener independientes los metadatos estructurados de la documentación técnica, aislar la interactividad del cliente del contenido renderizado en servidor y diferenciar la información independiente del idioma del contenido traducido permitió construir una arquitectura capaz de evolucionar sin requerir grandes reescrituras.

El proyecto también demostró que el desarrollo incremental no solo reduce riesgos de implementación, sino que permite descubrir mejores abstracciones.

Funcionalidades como la generación automática de navegación, el modelo de contenido bilingüe y la conservación de posición de lectura fueron incorporadas únicamente después de comprobar que sus versiones manuales resolvían una necesidad real.

Como resultado, cada abstracción fue creada alrededor de requisitos concretos y no de posibles problemas futuros.

Finalmente, quizá el resultado más importante del proyecto es que valida su propia filosofía.

Technical Journal no es solamente una plataforma utilizada para documentar trabajo de ingeniería; también es documentada siguiendo el mismo proceso que promueve para otros proyectos.

Cada decisión arquitectónica relevante, cada problema encontrado, cada compromiso asumido y cada aprendizaje obtenido forman parte del historial del proyecto en lugar de desaparecer después de implementar una funcionalidad.

En ese sentido, la plataforma nunca está realmente terminada.

Cada nueva mejora no representa únicamente una actualización del software, sino otro capítulo dentro de su propia historia de ingeniería, asegurando que la documentación continúe evolucionando junto con el sistema que describe.
