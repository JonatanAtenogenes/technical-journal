# IBM HR Analytics — Employee Attrition Analysis

## Introduction

Employee attrition is one of the most common challenges faced by Human Resources departments. Losing experienced employees affects productivity, increases recruitment costs, and often reveals deeper organizational issues that are not immediately visible.

This project explores the IBM HR Analytics Employee Attrition dataset with the objective of identifying patterns associated with employee turnover through a structured data analysis workflow.

Rather than focusing only on producing visualizations, the project is designed to demonstrate the complete analytical process—from understanding the business problem to communicating actionable findings. Every stage is documented as part of this case study, allowing the reader to follow both the technical implementation and the reasoning behind each decision.

The project also serves as a practical exercise in combining software engineering practices with data analysis. Instead of relying on a single notebook containing every step, reusable functionality is separated into Python modules while notebooks remain focused on business questions and analytical reasoning.

---

# Business Context

A company's Human Resources department has observed a growing number of employees voluntarily leaving the organization. Although demographic information, job characteristics and employee satisfaction metrics are available, there is no clear understanding of which factors are most strongly associated with attrition.

Without this knowledge, decisions regarding employee retention become reactive rather than proactive.

The purpose of this analysis is not to build a predictive machine learning model, but to understand the existing data, identify meaningful relationships between variables, and communicate findings that could support future business decisions.

This case study approaches the problem from the perspective of a Data Analyst working within an organization, where understanding the business problem is just as important as writing code.

---

# Business Questions

The project begins with a series of questions provided by the business rather than with technical requirements.

Among the questions explored are:

- What is the overall employee attrition rate?
- Which departments experience the highest employee turnover?
- Which job roles are more frequently associated with attrition?
- Does overtime appear to increase the likelihood of employees leaving?
- Is salary related to employee retention?
- Does job satisfaction influence attrition?
- What characteristics describe employees who are most likely to resign?

These questions guide every subsequent stage of the project, from data preparation to SQL analysis and exploratory visualizations.

Rather than exploring the dataset without direction, every notebook is designed to answer one or more of these business questions.

## Loading the Dataset

The analysis begins by loading the raw CSV file into a pandas DataFrame.

Rather than reading the file directly inside the notebook, the loading logic
is encapsulated within a reusable Python module located inside the `src`
package.

This keeps the notebook focused on analysis while making the loading process
reusable across future notebooks.

```python
from src.data import load_raw_data

df = load_raw_data()
```

The loading function itself remains intentionally simple.

```python
def load_raw_data() -> pd.DataFrame:
    return pd.read_csv(
        Path(__file__)
        .parent.parent
        .joinpath("data/raw/WA_Fn-UseC_-HR-Employee-Attrition.csv")
    )
```

Separating reusable functionality from analytical notebooks is a recurring
design decision throughout the project.

## Verifying Missing Values

Missing values frequently represent one of the largest sources of analytical
error.

For that reason, they are explicitly verified before any transformation is
performed.

```python
missing_values = (
    df.isnull()
      .sum()
      .sort_values(ascending=False)
)

missing_values = missing_values[missing_values > 0]
```

The dataset contains no missing values.

Because no imputation is required, later preprocessing stages can focus on
simplifying the dataset rather than repairing incomplete information.

---

## Searching for Duplicate Records

Duplicate employee records could bias subsequent analyses.

Verification is therefore performed before any calculations are made.

```python
df.duplicated().sum()
```

No duplicated records are detected.

This indicates that every row represents a unique employee observation.

---

## Identifying Constant Variables

Not every variable contributes useful information.

Some columns contain exactly the same value for every employee and therefore
cannot explain differences in attrition.

```python
unique_values = (
    df.nunique()
      .sort_values()
)

unique_values
```

Several variables contain only a single unique value.

These variables are identified during this stage and will be removed during
Data Preparation to simplify the dataset without losing analytical value.

## Why SQLite?

Instead of performing the entire analysis directly in pandas, the processed
dataset is loaded into a relational database.

Although SQLite is significantly simpler than production database systems
such as PostgreSQL, it allows the project to simulate a realistic analytical
workflow without introducing unnecessary infrastructure.

Using SQLite also provides an opportunity to demonstrate SQL as a primary
analytical tool rather than treating it as an optional addition.

The database therefore becomes an intermediate layer between data
preparation and exploratory analysis.
